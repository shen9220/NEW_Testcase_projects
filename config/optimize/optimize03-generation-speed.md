# 测试用例生成速度优化 — 根因分析与方案

## 问题

生成测试用例时，提取模块 → 逐模块生成整个流程耗时过长（典型 5 模块 PRD 需 4-5 分钟），用户体验差：
- 进度条永远卡在"提取功能模块"，用户以为系统卡死
- 无法取消正在进行的生成
- 逐模块生成严格串行，N 个模块 = N 次串行等待

## 根因分析

### 1. 进度回调缺失

`backend/app/routers/generation.py:36-39` 初始化 progress 后再无任何代码更新它。`AIGenerator.generate()` 是完整黑盒，内部 8 个 Skill 的执行对外部完全不可见。

前端 Steps 组件（`Workbench.tsx:276-286`）有完整的 8 步 UI，但 `currentSkillIndex` 始终为 0。

### 2. 逐模块生成严格串行

`AIGenerator.generate():183` 的 for 循环逐个 `await self._run_skill(...)`，N 个模块 = N 次串行 LLM 调用。

### 3. 独立 Skills 也串行

Skill 3 (约束提取)、Skill 5 (状态转换) 都只依赖 PRD 文本，互不依赖，但代码中仍串行执行。

### 4. 无取消/超时机制

- FastAPI BackgroundTasks 无内置超时
- 无取消端点
- LLM 调用单次超时 300s × 3 次重试 = 每个 Skill 最多 900s

### 5. LLM 调用次数统计

| 场景 | LLM 调用次数 | 估算耗时 |
|------|-------------|---------|
| 5 模块、首次通过、无显式约束 | 11 | ~4-5 分钟 |
| 5 模块、部分重试 | 14-18 | ~6-8 分钟 |
| 最坏情况（全部重试 + 3 轮校验） | 26 | ~13 分钟+ |

## 优化方案

### Hotfix (即时修复)

| 措施 | 效果 | 风险 |
|------|------|------|
| 进度回调机制 | Steps 实时推进，消除"卡死"假象 | 低 |
| 取消端点 | 用户可随时中断 | 低 |
| 服务端超时 (10min) | 防止无限挂起，保存部分结果 | 低 |

### Feature (性能优化)

| 措施 | 效果 | 风险 |
|------|------|------|
| 并行模块生成 (Semaphore(3)) | 5 模块从 150s → 60s | 中（API 限流） |
| 并行 Skill 3+5 | 节省 ~30s | 低 |
| 合并 Skill 3+4 | 减少 1 次 LLM 调用 | 低 |
| Skill 5 条件化 | 无状态 PRD 跳过此 Skill | 低 |
| PRD hash 缓存 | 相同 PRD 跳过 Skill 1 | 低 |

### 预期提升

- 典型 5 模块生成时间：4-5 分钟 → 1.5-2 分钟（**60%↓**）
- LLM 调用次数：11 → 6-7（**40%↓**）
- 用户体验：从"假死"到"清晰可追踪"
