# 优化需求

1、日志记录需要记录每次操作记录;成功，失败，每次点击都要记录
2、生成的测试用例按 prd 需求单个展示，可以点击二次展示 详细list，（支持按 prd 层级直接下载）
3、PRD 解析预览与生成的测试用例模块位置重叠，两个位置要分开展示，左右各占一块
4、用例生成后，增加一个模块覆盖率的统计，展示已覆盖模块数量/总模块数量
5、增加一个功能，用户可以选择只生成未覆盖模块的测试用例，针对未覆盖模块单独触发生成，直到覆盖率达到100%
6、生成的测试用例中，增加一个字段，标注该用例是否覆盖了 PRD 中的某个模块，如果没有覆盖，标注为“未覆盖”，并在 notes 字段中注明“需与 PM 确认：缺少 xxx 信息”，以便后续跟进补充。
7、生成的测试用例，分页加一个数量控制，默认每页展示 10 条，用户可以选择每页展示 20 条或 50 条。

# 优化分析  
1. **PRD 解析不完整**  
   - 现有解析库对 PDF/Word 的支持有限，尤其是表格和层级结构，导致模块信息提取不全。  
   - 图片内容完全被忽略，而 PRD 中的截图往往包含重要的功能说明。
2. **日志记录缺失**  
   - 操作日志记录不完善，缺少对用户操作和系统行为的详细记录。
3. **测试用例展示不友好**  
   - 测试用例展示方式单一，无法直观展示 PRD 的层级结构。
4. **覆盖率统计缺失**  
   - 缺乏对已覆盖模块数量的统计，无法直观了解测试覆盖情况。
5. **生成条件单一**  
   - 仅支持生成全部模块的测试用例，无法根据覆盖情况生成特定模块的测试用例。
6. **用例标注缺失**
    - 生成的测试用例中缺乏对覆盖情况的标注，无法区分哪些用例覆盖了 PRD 中的模块。
7. **分页展示缺失**
    - 测试用例展示没有分页功能，导致当测试用例数量较多时，用户体验较差。
    
# 优化方案
1. **优化 PRD 解析**  
   - 使用更强大的解析库，如 PDFMiner 或 OCR 技术，提高对 PDF/Word 文档的解析能力，特别是表格和层级结构的解析。
   - 对图片内容进行 OCR 识别，提取图片中的文字信息，补充到解析结果中。
2. **完善日志记录**  
   - 在用户操作的关键节点增加日志记录，如文件上传、解析、用例生成等。
   - 记录详细的操作信息，包括用户 ID、操作时间、操作类型、操作结果等。
3. **改进测试用例展示**  
   - 采用树形结构展示测试用例，按照 PRD 的层级结构进行展示，用户可以点击展开查看详细信息。
   - 支持按 PRD 层级直接下载测试用例，方便用户使用。
4. **增加覆盖率统计**  
   - 在生成测试用例后，统计已覆盖模块数量和总模块数量，并在界面上展示覆盖率信息。
5. **支持按覆盖情况生成测试用例**  
   - 增加一个功能，用户可以选择只生成未覆盖模块的测试用例，针对未覆盖模块单独触发生成，直到覆盖率达到100%。 
6. **增加用例标注**
   - 在生成的测试用例中增加一个字段，标注该用例是否覆盖了 PRD 中的某个模块，如果没有覆盖，标注为“未覆盖”，并在 notes 字段中注明“需与 PM 确认：缺少 xxx 信息”，以便后续跟进补充。
7. **增加分页展示功能**
   - 在测试用例展示页面增加分页功能，默认每页展示 10 条测试用例，用户可以选择每页展示 20 条或 50 条，以提升用户体验。

# 优化分析（问题根源）
1. **生成流程过于单一，缺乏模块化处理**  
   - 目前的生成流程是一次性处理整个 PRD，导致任何信息缺失都会影响整个流程，且无法针对具体模块进行补救。
2. **缺乏对生成结果的完整性校验**           
   - 生成完成后没有对输出的用例模块分布进行校验，无法发现遗漏的模块。
3. **过于依赖输入文本的完整性**  
   - 生成逻辑中存在“信息不足即跳过”的规则，导致任何描述不清的模块都被直接忽略，而不是生成带有“待确认”标记的用例。
4. **模型容量和输出长度限制**  
   - 现有模型的上下文长度限制可能导致输入文本被截断，从而导致信息丢失和生成结果不完整。
5. **Prompt 指令不够明确**  
   - 现有的 Prompt 指令没有明确要求覆盖所有模块，也没有要求在信息不足时生成带有标注的用例，导致模型倾向于跳过而不是补救。
6. **缺乏用户反馈机制**  
   - 生成结果没有提供覆盖率统计和针对未覆盖模块的二次生成功能，无法让用户主动参与到覆盖率提升的过程中。
