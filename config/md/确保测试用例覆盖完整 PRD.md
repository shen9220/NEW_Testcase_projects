# 优化方案：确保测试用例覆盖完整 PRD

## 问题现象
当前工具在生成测试用例时，只能覆盖 PRD 的部分内容，部分功能模块或需求点未输出对应测试用例。

## 原因分析
1. **文档解析不全**  
   - Word/PDF 解析时可能丢失非标题正文、表格、图片中的需求信息。  
   - Markdown 中的嵌套列表、折叠内容未被完全提取。

2. **上下文长度不足**  
   - 长篇 PRD 直接送入模型，超过最大上下文窗口，内容被截断。  
   - 单次生成 `max_tokens` 过小，导致输出 JSON 被截断，部分模块丢失。

3. **生成流程未强制覆盖所有模块**  
   - `extract_modules` 提取的模块列表可能被后续生成环节忽略。  
   - 某些模块因“信息不足”被直接跳过，而不是生成带有“待确认”标记的用例。

4. **模型输出不完整时无校验**  
   - 生成的用例模块分布未与模块列表进行比对，导致遗漏未能被发现并补救。

5. **分片逻辑不当**  
   - 按字数硬切可能导致模块被割裂，某个模块的需求散落在两个分片中，无法被完整识别。

---

## 优化方案（按实施优先级排序）

### 1. 增强文档解析，确保信息完整提取
- **PDF/Word**：升级解析库，保留表格内容和层级结构，转换为标准 Markdown。  
- 解析后增加人工复核可见性：前端预览页应展示完整的 Markdown 内容，便于确认解析无误。  
- 对于图片，若 PRD 依赖截图说明，可在解析结果中保留图片链接，并提示用户补充文本描述。

### 2. 采用“模块分治，分批生成”策略
- 将生成流程改为三步走：  
  **步骤一：全局模块识别**  
  调用 `extract_modules` 获取完整的模块列表（仅需模块名称、功能描述，内容很短，绝无截断风险）。  
  **步骤二：按模块逐一生成用例**  
  针对每个模块，提取 PRD 中与该模块相关的段落（可基于标题定位），分别调用 `generate_for_module`，确保每个模块都能产出用例。  
  **步骤三：边界和异常跨模块统一补充**  
  对涉及系统级异常（如网络、权限）的用例，可依据全局规则统一生成，但仍需对应到具体功能模块。
- 这样单次请求的输入和输出都很小，完全避开 token 限制，且任何模块都不会被遗漏。

### 3. 消除“需求残缺即跳过”的副作用
- 修改生成原则：**对于每个识别出的模块，必须至少生成一条基础用例**。  
- 若某个字段、按钮、状态因 PRD 描述不清无法确定，用例照常生成，但在 `notes` 字段明确标注“需与PM确认：缺少xxx信息”。  
- 这保证了覆盖率，同时将需求缺陷暴露给团队，而不是悄无声息地忽略。

### 4. 添加输出完整性校验与自动补全
- 生成完所有模块用例后，后端执行交叉比对：  
  提取的模块总数 vs 生成的用例中出现的模块数，若不一致，则对缺失模块重新调用生成（可更换措辞 prompt 以避免重复错误）。  
- 若某模块多次生成失败，记录到日志并输出警告，但依然保留已生成的其他用例。

### 5. 提升模型容量和输出长度
- 优先选用支持 **128K 上下文** 的模型（如 `deepseek-chat` 当前版本），可直接输入全文。  
- 将 `generate_for_module` 的 `max_tokens` 设置为 4096，确保即便一个模块用例很多也不会截断。  
- 同时开启 `finish_reason` 检测，若为 `length` 则自动扩大 `max_tokens` 重试。

### 6. 优化 Prompt 指令，要求全覆盖
在 `system_prompt` 中加入：
```text
你必须对以下每个功能模块都生成至少一条测试用例，即使信息不完整也要通过notes标注。
输出用例列表时，保证所有模块均被包含，模块名必须与“待测模块列表”完全一致。

### 7. 引入用户反馈机制（可选）
生成结果页面展示“已覆盖模块 / 总模块数”的统计。
用户可勾选“只生成未被覆盖的模块”，针对缺口单独触发二次生成。

*** 推荐实施路径 ***
1.先修改生成流程为“分模块串行生成”，这是解决遗漏最根本的方法。
2.同步修改 Prompt 和去跳过逻辑，保证每个模块至少有一条用例。
3.提升单次输出容量，避免截断。
4.增加比对校验，确保万无一失。
