# PRD 处理与测试用例生成常见错误及综合解决方案

## 当前错误分析

### 错误现象


### 根本原因
后端代码中某处调用了 `.get()` 方法，但目标对象实际为字符串，而非预期的字典或类似结构。高频发生位置：

1. **PRD 解析结果使用错误**  
   - 解析模块返回了纯文本字符串，而后续代码将其当作字典对象访问（例如 `parsed_prd.get("modules")`）。
   - 应统一为结构化对象，比如 `parsed_prd` 应为 `{"modules": [...], "full_text": "..."}`。

2. **LLM 返回内容解析失败**  
   - Skill 输出 JSON 解析为字符串后，代码假定是字典而直接调用 `.get()`。
   - 实际 JSON 解析可能因为格式错误被捕获后返回原始字符串，或未做解析直接传递。

3. **API 响应反序列化问题**  
   - 从文件或 API 读取的 JSON 数据被多次序列化/反序列化，导致某些本应是对象的字段变成了字符串。

4. **配置读取错误**  
   - 某个配置项期望字典，但 YAML/JSON 中写成了字符串（例如 `bot: "testcase_generator"` 而代码期望 `bot.get("name")`）。

## 综合解决方案

### 1. 强化数据结构定义与校验
- 在代码中为 PRD 解析结果、LLM 输出、配置等定义明确的 Pydantic 数据模型。
- 所有关键边界处使用模型校验，无效数据直接拦截并给出清晰错误。
- 禁止裸字符串在需要字典的上下文中传递，发现即抛出类型错误。

### 2. 实施防御性 JSON 处理
- 所有 `json.loads()` 调用用 try-except 包裹，失败时立即记录原始数据并返回标准错误体。
- 使用 `json_repair` 作为一级修复手段，修复后再解析。
- 对 LLM 返回内容进行预处理：去除 Markdown 代码块标记、提取首个 JSON 对象。

### 3. 完善文件解析器
- 为 PDF、Word、Markdown 解析器编写单元测试，覆盖常见排版变异。
- PDF 解析增加表格检测（`camelot` 或 `tabula`）并将表格转换为 Markdown 表格。
- Word 解析保留样式信息，基于样式名重建层级。
- 解析完成后统一返回 `{"text": "...", "structure": {...}}` 结构。

### 4. 优化 AI 调用鲁棒性
- 固定使用 `max_tokens` ≥ 4096，并根据 `finish_reason` 动态调整。
- 建立两级 Bot：主 Bot 失败或超时后自动切换到备用 Bot（已实现但需确保配置正确）。
- 在调用前自动计算 token 数，若超过模型限制则自动分片。
- 所有网络错误进行自动重试（3次指数退避）。

### 5. 用例生成覆盖保证
- 先提取完整模块列表，再逐个模块生成，生成后交叉比对，缺失模块立即补全。
- 若某模块连续 3 次生成失败，标记为“需求不足”并停止尝试，在前端展示详细失败原因。
- 放宽校验：不因模糊词直接丢弃用例，改为标注 `review_required` 字段。

### 6. 统一错误处理中间件
- 后端全局捕获所有异常，返回标准化错误响应：
  ```json
  {
    "status": "error",
    "error_type": "JSON_PARSE_ERROR",
    "detail": "...",
    "suggestion": "请检查PRD内容或重试"
  }
  前端根据 error_type 展示对应提示和操作建议。

### 7. 数据存储防错
使用 pathlib 创建目录并检查权限。
文件名加入毫秒级时间戳和随机数防止重名。
JSON 序列化时使用自定义编码器，将不可序列化对象转为字符串。

### 8. 端到端自测用例
建立一条自动化测试流水线，包含：
上传一个已知完整的 PRD 样本，验证是否所有模块都生成用例且无臆想。
上传一个故意残缺的 PRD，验证是否返回失败信息和缺失标注。
检查日志中无未捕获的 AttributeError。

通过以上系统性加固，不仅能解决当前 'str' object has no attribute 'get' 错误，还可预防和自修复绝大多数因数据格式、网络、模型输出不稳定导致的生成中断或覆盖不全问题。
