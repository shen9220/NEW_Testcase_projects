# 深层分析：为何补全后仍无法100%覆盖测试用例

## 问题描述
即使实施了分模块生成策略，点击“补全未覆盖模块”后，某些模块依然没有生成出可用用例，导致无法达到100%覆盖。反复补全也无效。

---

## 深层原因分析

### 1. 模块提取本身就不完整
- `extract_modules` 同样依赖 AI，它可能遗漏部分模块，尤其是：
  - 隐含需求（例如 PRD 只说“支持导出”，没单独列出标题，AI 可能忽略）。
  - 分散描述的功能（同一模块在 PRD 多处提及，但未形成独立段落）。
- **后果**：补全列表里根本没有这些模块，自然无法生成。

### 2. 分片或上下文供给不当
- 即使分模块生成，传给 AI 的 **上下文可能只包含该模块的片面片段**，而不是全局 PRD 视图。
- 如果 PRD 中模块 A 的规则引用模块 B 的定义，但生成 A 时没有提供 B 的信息，AI 会因信息不足而返回空或低质量结果。
- 补全时仍是同样片段，重复失败。

### 3. 补全请求的结构限制
- 补全接口可能只允许小批量生成，且 prompt 设计过于简单（如“请为模块X生成用例”），导致 AI 再次因信息不全或指令不明确返回失败。
- 若补全使用与原生成相同的 `temperature` 和 prompt，相同输入可能得到同样失败结果。

### 4. 去重/校验机制错误地丢弃了已生成用例
- 某些用例生成后，质量校验器可能将其判定为“无效用例”（如预期结果太模糊），但未记录在案，反而被删除。
- 补全时该模块显示为“未覆盖”，重新生成仍可能再次被过滤，进入死循环。

### 5. 硬性跳过规则过于严格
- 为防臆想，设定了“无UI元素名不生成”、“无明确规则不生成”等规矩。某些模块的 PRD 描述极其概括（如“用户管理页面可查看列表”），完全找不到可测试的步骤，系统拒绝生成。
- 这本身符合规范，但这类模块在补全列表中永远无法被覆盖，用户期望有相应提示。

---

## 解决方案

### A. 确保 extract_modules 的完整性
- 采用**多轮提取 + 合并**策略：
  - 第一次：按标题层级提取模块。
  - 第二次：提取所有动词短语（如“查看”、“导出”、“编辑”），补充隐含模块。
- 在提取后，让用户在前端确认模块列表，可手动添加遗漏项。形成最终模块清单。

### B. 改进模块用例生成的上下文组装
- 生成某个模块时，不仅给该模块对应的 PRD 段落，还要**附带全局的角色定义、公共规则、异常处理章节**。
- 使用 Prompt 格式：
【全局信息】
角色：... 公共规则：...
【当前模块】
...模块段落...
【任务】：生成该模块的测试用例。

### C. 补全流程专项优化
- 补全请求使用**更宽松的生成条件**：
- 临时将 `temperature` 调高到 0.3，增加变化性。
- 在 prompt 中加入原失败原因与要求：“即使信息不全，也必须生成至少一条用例，缺失信息标注在notes中”。
- 补全时允许输出低优先级用例，优先保证覆盖率。

### D. 建立“无法生成”模块的标记机制
- 若某模块连续尝试 3 次仍无法产出有效用例（可能是 PRD 质量太差），**标记该模块为“需求缺失”**，而非反复尝试。
- 在前端展示带有警告图标的模块，并给出理由，用户手动确认或补充信息后再触发生成。

### E. 调整校验规则，避免误杀
- 校验器只拦截明显违反规则的用例（如重复完全相同步骤），不要因模糊词直接丢弃。
- 模糊标注留在 `notes` 中供人工审核，而非删除用例。这样能保证覆盖率。

### F. 引入模块级状态追踪
- 在 JSON 记录中为每个模块维护生成状态：`covered` / `partial` / `failed`。
- 补全按钮只重新生成状态为 `failed` 或 `partial` 的模块，并携带详细失败日志，便于定位。

---

## 实施后的预期效果
- 模块提取完整度 ≥ 95%
- 覆盖率达到 100%（即使某些模块的用例质量较低，也会生成，并由标注暴露风险）
- 补全按钮最多点击一次即可解决遗漏，连续失败的情况被标记为“需求缺失”停止重试

按照以上方案优化后端生成逻辑与前端模块确认流程，即可解决补全仍遗漏的痛点。
