# 测试用例生成规范与防臆想机制

> 文档版本：v1.1
> 目标：杜绝臆测、固化输出，确保生成的用例严格基于PRD且可执行，失败即暴露需求缺陷。
> 本文档定义的是**原则与机制**，具体实现见 [AI配置规范.md](AI配置规范.md) 的 7+1 Skill 流水线。

---

## 1. 核心生成原则

**零臆想、精准标注缺失**

- AI 仅基于 PRD 中**明确描述**或**可严格推导**的信息生成用例。
- PRD 有明确约束的字段 → 可生成边界用例；PRD 无约束 → 不猜测，在用例备注中标注"需与产品确认：xxx"。
- 若 PRD 信息严重不足（无法提取任何有效功能点），返回详细失败原因，不输出任何用例。
- 禁止使用预置模板填充——每次生成必须是**动态解析-推理-构建**的新结果。同一份 PRD 多次生成，核心验证点一致但表述可有差异。

**测试工程师分析思维**

生成用例时模拟真实测试人员的分析过程：
1. 识别功能模块与输入/输出
2. 识别显式规则与隐式约束
3. 梳理状态变化与转换条件
4. 寻找边界条件与异常分支

## 2. 防止臆想的技术保障

### 2.1 严格的信息来源控制（Prompt 层）

在 System Prompt 中强制执行信息来源约束：

- **可生成用例的信息必须满足以下之一：**
  1. PRD 原文直接陈述的 UI 元素、字段、流程
  2. PRD 中表格/字段定义的数值类型、长度、必填等显式约束
  3. 由明确规则**严格推导**的路径（如"点击提交"必存在"提交"按钮）

- **绝对禁止使用以下信息：**
  1. 产品通用常识（如"通常有忘记密码功能"）——除非 PRD 写明
  2. 其他系统的参考实现
  3. 测试人员的经验性补充

- **强制检查指令**（嵌入每个 Skill 的 system prompt）：

> 在生成每一步 action/expected 前，确认该步骤在 PRD 中有对应的文字依据。如果答案是否，舍弃该步骤。

### 2.2 结构化中间表示（Intermediate Representation）

AI 先将 PRD 转为结构化中间表示，再基于该表示生成用例。该中间表示由 Skill 1（extract_modules）输出。

**中间表示格式（JSON Schema，与 Skill 1 输出对齐）**：

```json
{
  "modules": [
    {
      "name": "用户登录",
      "summary": "用户通过手机号验证码完成登录",
      "relevant_text": "（PRD 原文段落）",
      "ui_elements": ["手机号输入框", "获取验证码按钮", "登录按钮"],
      "fields": [
        {"name": "手机号", "type": "string", "length": 11, "format": "中国大陆手机号"}
      ],
      "actions": ["点击获取验证码", "输入验证码点击登录"],
      "rules_explicit": [
        "手机号格式错误时提示",
        "验证码60秒后可重发",
        "手机号已注册则跳转"
      ],
      "states": ["未登录", "登录中", "已登录"],
      "missing_info": ["验证码有效期未说明"]
    }
  ]
}
```

**约束**：
- `ui_elements` 为空 → 该模块无法生成可执行步骤，跳过并记录缺失
- `missing_info` 中的项必须出现在最终生成的用例备注中

### 2.3 输出后验证器（Sanity Checker）—— Skill 8

在 AI 完成去重（Skill 7）后，后端运行**规则校验器**（Skill 8: post_validation），不通过则打回修正，最多重试 2 次，仍失败则终止并报告。

| 校验项 | 规则 | 不通过处理 |
|--------|------|-----------|
| 模糊词检测 | 预期结果中若出现"正常""正确""无误""成功"且无具体界面变化描述 | 打回要求具体化 |
| 元素来源检测 | 步骤中的 UI 元素名是否可追溯到 PRD 原文或中间表示 | 标记可疑，强制在备注中标注需求缺失 |
| 数值合理性 | 边界用例中的数值必须能从 PRD 字段约束中确定或计算 | 不允许随机边界值，打回修正 |
| 步骤可达性 | 每条步骤的 action 能对应到明确的前置界面/状态 | 无法到达的步骤打回 |

**校验器实现要点**（详见 [开发规范.md](开发规范.md)）：

```python
# 伪代码
async def post_validation(testcases: list, intermediate_rep: dict) -> ValidationResult:
    issues = []
    for tc in testcases:
        # 1. 模糊词检测
        vague = detect_vague_terms(tc["steps"])
        if vague:
            issues.append({"case_id": tc["case_id"], "type": "vague", "detail": vague})
        # 2. 元素来源检测
        unknown_elements = check_element_sources(tc["steps"], intermediate_rep)
        if unknown_elements:
            issues.append({"case_id": tc["case_id"], "type": "unknown_element", "detail": unknown_elements})
        # 3. 数值合理性
        invalid_boundary = check_boundary_values(tc, intermediate_rep)
        if invalid_boundary:
            issues.append({"case_id": tc["case_id"], "type": "invalid_boundary", "detail": invalid_boundary})
        # 4. 步骤可达性
        unreachable = check_step_reachability(tc["steps"], intermediate_rep)
        if unreachable:
            issues.append({"case_id": tc["case_id"], "type": "unreachable_step", "detail": unreachable})
    return {"passed": len(issues) == 0, "issues": issues}
```

### 2.4 模块驱动的动态上下文

- 不同模块的生成 prompt 动态组装，包含该模块相关的 PRD 原文片段
- 不使用"测试用例通用检查点"之类的固定列表——从需求本身提取检查点

## 3. 防止用例固定化的设计

### 3.1 基于语义而非模板的生成

- Prompt 中**禁止给出内容模板**（如"请按以下用例格式输出：{示例}"），这会诱导 AI 模仿内容
- **允许给出 JSON Schema 结构约束和字段说明**（如"每步需包含 action 和 expected 字段"）
- 要求 AI 自行决定步骤数量和描述方式，鼓励差异性

### 3.2 温度参数控制

- 全局使用低温度 0.1-0.15，保证格式约束和逻辑一致性
- 不在单次生成中混用不同温度值（避免输出不可控）
- 每次生成时在 metadata 中注入 task_id + 时间戳，避免 LLM 缓存导致完全重复

### 3.3 上下文隔离

- 每次生成任务完全独立，不将历史生成结果作为 Few-Shot 示例注入 LLM 上下文
- 禁止使用"常见测试用例库"作为参考——LLM 只能基于当前 PRD 进行分析
- 用户在前端标记的"臆想/无用"用例及修正版本，**收集到反馈库中，由开发者人工分析后优化 Skill Prompt，而非自动注入 LLM**

## 4. 生成管道：概念框架 → Skill 流水线映射

### 概念框架（四阶段分析模型）

```
[PRD全文]
   │
   ▼
[阶段1: 功能点提取] → 模块、字段、动作、规则、状态
   │  校验: 是否有明确输入输出？→ 无则标记缺失，无任何模块则终止
   ▼
[阶段2: 路径枚举] → 正常路径 + 异常分支（仅基于规则推导，不臆想）
   │
   ▼
[阶段3: 用例组装] → 路径→用例，填入具体字段值，预期结果需可量化
   │
   ▼
[阶段4: 后处理校验] → 模糊词/元素来源/数值合理性/步骤可达性
   │  失败 → 重试(最多2次) → 仍失败则返回错误详情
   ▼
[最终用例列表]
```

### 映射到 7+1 Skill 流水线

| 概念阶段 | 对应 Skill（详见 AI配置规范.md） | 说明 |
|----------|-------------------------------|------|
| 阶段1: 功能点提取 | **Skill 1: extract_modules** | 提取模块列表 + 中间表示（含 ui_elements/fields/actions/rules/states/missing_info） |
| 阶段1: 字段约束 | **Skill 3: extract_constraints** | 补充提取字段级约束 |
| 阶段2: 路径枚举 | **Skill 2: generate_for_module** | 逐模块生成正向+异常路径用例 |
| 阶段2: 异常分支 | **Skill 4: boundary_completion** + **Skill 5: state_transition** | 边界补充 + 状态转换补充 |
| 阶段3: 用例组装 | **Skill 6: refine_steps** | 步骤具体化，确保可执行 |
| 阶段3: 去重 | **Skill 7: deduplicate** | 语义去重 |
| 阶段4: 后处理校验 | **Skill 8: post_validation** (Sanity Checker) | 规则校验，不通过打回修正 |

## 5. "需求残缺即终止"策略

### 终止条件

- 阶段 1（Skill 1）后检查：若**没有任何模块**能提取到有效的 UI 元素或明确输入输出 → 终止生成
- 部分模块残缺：跳过该模块，在最终结果中汇总"因需求不明确而跳过"的模块列表
- 全部模块残缺：返回失败，不输出任何用例

### 成功/失败判定

**生成成功条件**：至少为 PRD 中 1 个模块生成 ≥1 条通过 Skill 8 校验的用例。

**生成失败条件**：没有任何模块能生成通过校验的用例。

### 失败反馈格式（对齐 API 标准响应）

在 `GET /generation/{task_id}/status` 中返回：

```json
{
  "code": 200,
  "data": {
    "task_id": "20260509-143022",
    "status": "failed",
    "testcase_count": 0,
    "failure": {
      "reason": "generation_failed",
      "details": [
        "用户登录模块：缺少密码格式要求，无法生成密码边界用例",
        "订单支付流程：无任何支付方式说明，无法生成正向用例"
      ],
      "skipped_modules": ["订单支付"],
      "suggestion": "请补充上述需求细节后重新生成"
    },
    "completed_at": "2026-05-09T14:32:10"
  }
}
```

> 不会为了"保证有输出"而填充通用用例（如"检查页面加载正常"）。

## 6. 持续改进机制

- 用户在前端可标记某条用例为"臆想/无用"，并将修改后的版本提交
- 后端收集被标记的用例与修正版本，存储到 `data/feedback/` 目录
- 开发者定期分析高频缺失点和臆想模式，**优化 Skill Prompt（.yaml 配置文件），而非增加兜底模板**
- 提示词采用配置化（YAML），开发者可基于失败反馈快速迭代

## 7. 开发与测试验证

单元测试覆盖以下场景（详见 [开发规范.md](开发规范.md) 测试规范章节）：

| 场景 | 输入 | 期望输出 |
|------|------|----------|
| 完整约束 PRD | 包含字段类型/长度/规则的完整 PRD | 生成符合数量要求的用例，覆盖正向/异常/边界 |
| 故意模糊 PRD | 仅有功能描述，无字段约束 | 返回失败或标注大量"需与产品确认"，不杜撰约束 |
| 多次生成相同 PRD | 同一 PRD 生成 3 次 | 核心验证点相同，但用词/步骤描述不完全一致 |
| 臆想拦截 | PRD 仅描述手机号登录 | 不应出现"用户名密码登录""邮箱登录"等用例 |

集成测试确保 Skill 8（Sanity Checker）能拦截常见臆想，且修正重试机制正常工作。
