# 真正可操作的测试用例与错误用例示范

## 示例 PRD 片段

以下 PRD 片段将用于演示正确和错误的用例生成：

> **用户注册模块**
>
> 用户输入手机号（11位中国大陆手机号，以1开头）和密码（6-20位，必须包含字母和数字）。
> 点击"注册"按钮。系统发送短信验证码到该手机，页面弹出验证码输入框（4位数字验证码，有效期60秒，可重新发送）。
> 用户输入验证码并点击"确认"，完成注册，自动跳转到首页。
>
> 约束：
> - 手机号已在系统中注册时，点击注册按钮后提示"该手机号已注册"
> - 同一手机号每天最多发送5次验证码
> - 网络异常时，提示"网络开小差，请稍后重试"
> - 验证码错误时，提示"验证码错误，请重新输入"，剩余尝试次数3次
> - 验证码过期时，提示"验证码已过期，请重新获取"

## 错误示范（不可执行 / 臆想）

### ❌ 错误1：步骤模糊，无可操作数据

```json
{
  "action": "输入正确的手机号和密码",
  "expected": "注册成功"
}
```

**问题**：
- 未指明具体的测试数据（"正确的"是多少？）
- "注册成功"无具体观察点（页面跳转？提示文字？URL变化？）
- UI元素不明确（哪个输入框？）

**正确做法**：写明具体值 `'13912345678'`，预期写明具体现象 `提示'注册成功'，跳转到 /home`

### ❌ 错误2：臆想 PRD 不存在的功能

```json
{
  "title": "使用邮箱注册",
  "steps": [{"action": "点击邮箱注册Tab", "expected": "切换到邮箱注册表单"}]
}
```

**问题**：PRD 中仅提及手机号注册，没有邮箱注册功能。该类用例完全是测试人员臆想。

**预防机制**：Skill `extract_modules` 只能提取 PRD 中实际存在的模块，不存在的功能不会有对应模块，也就不会生成用例。

### ❌ 错误3：预期结果不可量化

```json
{
  "expected": "系统表现正常"
}
```

**问题**："正常"无法评判——没有人能判断什么算"正常"。应改为具体界面变化。

**预防机制**：Skill `refine_steps` 会检测模糊表述并具体化。

### ❌ 错误4：缺少异常/边界覆盖

如果只生成了正向用例（输入正确→注册成功），完全没有覆盖：手机号已注册、密码格式错误、验证码过期、网络异常等场景。

**预防机制**：Skill `generate_for_module` 的 system prompt 强制要求覆盖正向、异常、边界、状态转换四种类型。

### ❌ 错误5：步骤跳跃，缺少中间状态

```json
{
  "steps": [
    {"action": "输入手机号和密码", "expected": "完成注册，跳转首页"}
  ]
}
```

**问题**：跳过了"点击注册按钮"→"弹出验证码输入框"→"接收验证码"→"输入验证码"→"点击确认"这些中间步骤。

## AI 生成 vs 固化模板的区别

| 对比维度 | 固化模板方式 | AI Skill 流水线方式 |
|----------|-------------|-------------------|
| 模块识别 | 预设"登录/注册/首页"等固定模块 | 从 PRD 原文提取，PRD 有什么就识别什么 |
| 用例内容 | 填充模板变量 | 根据 PRD 具体描述生成，步骤引用实际 UI 元素名 |
| 边界值 | 固定边界值（如"输入超长字符串"） | 根据 PRD 中声明的约束动态确定（6-20位 → 测5/6/20/21位） |
| 缺失处理 | 模板留空或编造 | 标注"需与产品确认：PRD未说明xxx" |
| 异常场景 | 固定套路（网络超时、500） | 从 PRD 中识别具体的异常分支（如"同一手机号每天5次上限"） |
