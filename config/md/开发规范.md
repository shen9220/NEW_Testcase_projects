# 开发规范

## 一、总体原则

1. **代码分层清晰**：Router（参数校验）→ Service（业务逻辑）→ Repository（数据读写），每层只做自己的事
2. **单一职责**：一个文件/类只做一件事，功能变更时只需改一处
3. **依赖注入**：Service 和 Repository 通过构造函数接收依赖，方便单元测试 Mock
4. **异常处理**：只在 Repository 层捕获外部异常（文件IO、网络请求），Service 层抛出业务异常，Router 层转换为 HTTP 响应
5. **向后兼容**：数据模型增加字段用默认值，不删字段，JSON 数据格式保持向前兼容

## 二、后端规范 (Python / FastAPI)

### 2.1 项目结构规范

- `routers/` 层：只做请求参数校验、调用 Service、格式化响应，**不写业务逻辑**
- `services/` 层：所有业务逻辑，调用 Repository 读写数据
- `repositories/` 层：封装数据访问细节（本地文件 / Supabase），对外暴露统一接口
- `models/` 层：Pydantic 模型定义，与前端 TypeScript 类型保持一致

### 3.3 状态管理

不引入全局状态库，使用 React hooks + 组件状态即可：

```typescript
// hooks/useTestCases.ts - 自定义 hook 封装数据逻辑
export function useTestCases(taskId: string) {
  const [cases, setCases] = useState<TestCase[]>([]);
  const [loading, setLoading] = useState(false);
  const [pagination, setPagination] = useState({ page: 1, pageSize: 20, total: 0 });
  
  const fetchCases = useCallback(async (page = 1) => {
    setLoading(true);
    try {
      const res = await testcaseService.list(taskId, page, pagination.pageSize);
      setCases(res.data.items);
      setPagination(prev => ({ ...prev, page, total: res.data.total }));
    } finally {
      setLoading(false);
    }
  }, [taskId]);
  
  return { cases, loading, pagination, fetchCases, setCases };
}
```

### 3.4 错误边界

```tsx
// 每个页面包裹 ErrorBoundary，避免局部错误导致白屏
class ErrorBoundary extends React.Component<Props, { error: Error | null }> {
  state = { error: null };
  
  static getDerivedStateFromError(error: Error) {
    return { error };
  }
  
  render() {
    if (this.state.error) {
      return <ErrorFallback error={this.state.error} onRetry={...} />;
    }
    return this.props.children;
  }
}
```

## 四、AI 生成规范

### 4.1 Prompt 管理

- 所有 Skill Prompt 集中在 `backend/app/services/skill_prompts.py` 中管理
- 每个 Skill Prompt 作为模块级常量 `SKILL_EXTRACT_MODULES = """..."""`
- 修改 Prompt 时不需要改业务代码，只需改常量
- Prompt 配置文件支持 YAML 格式，方便非开发人员基于失败反馈快速调整

### 4.2 生成质量保证

async def execute_skill(skill_name: str, prompt: str, user_input: str) -> list:
    response = await llm.chat(system=prompt, user=user_input)
    
    # 1. 提取 JSON（LLM 可能包裹在 ```json 中）

def check_module_availability(intermediate_rep: dict) -> tuple[list, list]:
    """检查提取的模块是否可生成用例"""
    valid = []
    skipped = []

### 4.4 Sanity Checker 实现要点

```python
# backend/app/services/skill_executor.py

import re

# 模糊词检测正则
VAGUE_PATTERN = re.compile(
    r'(正常|正确|无误|成功|表现正常|符合预期)'
    r'(?!.*(?:提示|跳转|URL|页面|弹窗|Toast|按钮|输入框|颜色|文字|图标|消失|变化))'
)

def detect_vague_terms(steps: list) -> list:
    """检测预期结果中的模糊描述"""
    issues = []
    for i, step in enumerate(steps):
        if VAGUE_PATTERN.search(step.get("expected", "")):
            issues.append({
                "step_index": i,
                "expected": step["expected"],
                "suggestion": "描述具体的UI变化（如提示文字、页面跳转URL、按钮状态变化）"
            })
    return issues

def check_element_sources(steps: list, all_ui_elements: set) -> list:
    """检测步骤中引用的UI元素是否来自PRD"""
    issues = []
    for i, step in enumerate(steps):
        # 从 action 中提取可能的 UI 元素名（引号内的文本）
        mentioned = re.findall(r'[\'"「『]([^\'\"」』]+)[\'"」』]', step.get("action", ""))
        unknown = [e for e in mentioned if e not in all_ui_elements]
        if unknown:
            issues.append({
                "step_index": i,
                "unknown_elements": unknown,
                "suggestion": "修正为PRD实际提及的UI元素名称"
            })
    return issues
```

### 4.5 校验失败重试逻辑

```python
# 校验失败时，将 issues 反馈给 refine_steps 重试
MAX_VALIDATION_RETRIES = 2

async def generate_with_validation(all_cases, intermediate_rep, prd_text):
    for attempt in range(MAX_VALIDATION_RETRIES + 1):
        # 细化 + 去重
        refined = await execute_skill("refine_steps", ..., all_cases)
        deduped = await execute_skill("deduplicate", ..., refined)
        
        # Sanity Checker (不调用 LLM)
        validation = post_validation(deduped, intermediate_rep, prd_text)
        
        if validation["passed"]:
            return {"status": "completed", "testcases": deduped}
        
        if attempt < MAX_VALIDATION_RETRIES:
            # 注入校验失败信息到 refine_steps 的 user prompt
            all_cases = deduped  # 以去重结果为基础重试
        else:
            # 重试耗尽
            return {
                "status": "partial",
                "testcases": deduped,
                "validation_issues": validation["issues"]
            }
```

### 4.6 防臆想检查清单

每个 Skill 执行后自动检查：

| 检查项 | 规则 | 不通过处理 |
|--------|------|-----------|
| 模块名是否来自 PRD | 对每个模块名在 PRD 原文中搜索 | 标注 "未在PRD中找到" |
| 用例步骤是否有具体值 | 检测模糊词（"正确"/"有效"/"正常"） | 触发 Skill 6 重新细化 |
| 是否存在纯臆想功能 | 对模块名不在 extract_modules 结果中的用例 | 移入审核列表 |
| UI 元素可追溯 | Skill 8 校验步骤中的 UI 元素是否来自 PRD | 标记可疑，强制标注缺失 |
| 步骤可达性 | Skill 8 校验每步是否可从前往后到达 | 打回重新编排步骤顺序 |

## 六、Git 工作流

1. **主分支**：`main`
2. **功能分支**：`feature/{描述}`，如 `feature/file-upload`
3. **提交信息**：中文或英文均可，格式 `{类型}: {描述}`
   - `feat: 添加PRD文件上传功能`
   - `fix: 修复Excel导出时步骤丢失问题`
4. **提交前**：
   - 运行 `pytest` 确保后端测试通过
   - 运行 `npm run lint` 确保前端无 lint 错误
   - 确认 `.env` 不在提交中
