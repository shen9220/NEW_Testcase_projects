# 分支名称不匹配问题

## 错误现象

```bash
$ git push -u origin master
错误：源引用规格 master 没有匹配
错误：无法推送一些引用到 'github.com:shen9220/New_Testcase_Projects.git'
```

## 原因

Git 2.28+ 版本默认初始化分支名为 `main`，而旧版本和许多仓库默认使用 `master`。本项目本地分支为 `main`，但配置和远程期望的是 `master`，导致 `git push origin master` 找不到对应的本地分支。

## 解决方案 1：重命名本地分支

将本地分支重命名为 `master`：

```bash
git branch -M master
```

之后即可正常推送：

```bash
git push -u origin master
```

## 其他方案

如果想保持本地 `main` 分支名，推送到远程 `master`：

```bash
git push -u origin main:master
```

## 预防措施

在 `git init` 后立即指定默认分支名：

```bash
git init --initial-branch=master
```

或全局设置：

```bash
git config --global init.defaultBranch master
```

## 补充：Push 被拒绝 — 仓库规则违规

### 错误现象

```bash
$ git push -u origin master
To github.com:shen9220/New_Testcase_Projects.git
 ! [remote rejected] master -> master (push declined due to repository rule violations)
错误：无法推送一些引用到 'github.com:shen9220/New_Testcase_Projects.git'
```

### 原因

GitHub 的 **Secret Scanning（密钥扫描）** 机制检测到推送的提交历史中含有敏感信息（如 GitHub Personal Access Token）。当 `key.md` 文件中包含真实的 `github_token: ghp_xxx...` 并被提交到 Git 历史后，GitHub 的 Push Protection 会直接拦截推送，防止密钥泄露到远程仓库。

### 解决方法

**注意**：仅修改文件并重新 commit 是不够的，因为原始 token 仍然存在于历史 commit 中，GitHub 会扫描整个推送链。

正确步骤：

1. 将文件中的真实 token 替换为占位符（如 `<你的GitHub Token>`）
2. 暂存修改后的文件
3. 使用 `git commit --amend` 将修改合并到前一个 commit 中（或使用 `git rebase -i` 清理多个 commit）
4. 验证历史中不再包含 token：

   ```bash
   git show HEAD:github/git_config/key.md | grep token
   ```

   应只显示占位符而非真实 token

5. 重新推送

### 防范建议

- **永远不要**将真实 token/密码/密钥提交到 Git 仓库
- 配置文件使用占位符，通过环境变量或 `.env` 文件（加入 `.gitignore`）注入真实值
- 如果 token 曾经被推送过，立即在 GitHub 设置中撤销该 token 并生成新 token
