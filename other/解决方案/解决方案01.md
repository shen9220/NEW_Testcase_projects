# 原因：Skill extract_modules 输出格式错误，无法解析为JSON: Unterminated string starting at: line 255 column 24 (char 9373)

# 修复 LLM 输出 JSON 解析错误（Unterminated string）

## 错误现象
调用 Skill extract_modules 时，大模型返回的内容无法作为 JSON 解析，提示在某个位置存在未闭合的字符串。

---

## 解决方案

### 1. 增大 max_tokens 并检测截断
- 原因：输出 JSON 可能因为 max_tokens 设置过小而被中途截断，导致字符串不完整。
- 措施：将该 Skill 的 max_tokens 至少设置为 4096 或更高（如 8192）。同时检查 API 返回的 finish_reason 字段，若为 `length` 说明输出被截断，应自动提高 max_tokens 并重新请求。

### 2. 使用 JSON 自动修复工具
- 原因：大模型生成的 JSON 经常存在微小格式错误，可通过专门的修复库自动修正。
- 措施：在解析前，先用 `json_repair` 等库对输出文本进行修复，它能够自动补全缺失的引号、括号，处理未转义字符等。修复后再尝试用标准 JSON 解析器加载。

### 3. 强化 Prompt 格式约束
- 原因：模型可能输出了 Markdown 代码块标记（如 ```json）或文末多余解释，或者未对特殊字符进行正确转义。
- 措施：在 System Prompt 中明确要求：
  - 直接返回纯 JSON，不要用 Markdown 代码块包裹。
  - 所有字符串内部的双引号必须用反斜杠转义。
  - 避免在字段值中使用未转义的换行符，可以用空格替代。
  - 确保输出的最后一行是闭合的 `}` 或 `]`。
  - 如果内容太长可能被截断，请优先保证 JSON 结构完整，必要时精简描述。

### 4. 输出预处理与提取
- 原因：原始输出可能包含额外的非 JSON 文本，导致解析器找不到完整的 JSON 块。
- 措施：在解析前，对输出进行清洗：
  - 去除首尾空白。
  - 若发现以 ``` 开头，则剥离 Markdown 代码块标记。
  - 从文本中提取第一个完整的 JSON 对象（利用括号匹配算法），丢弃多余部分。
  - 若提取后的 JSON 仍不合法，再送入自动修复工具。

### 5. 拆分任务，降低单次输出复杂度
- 原因：如果 PRD 模块数量太多，一次输出的 JSON 会非常庞大，容易超过长度限制或引发截断。
- 措施：将提取功能点的大任务拆分成多次调用，例如按一级标题把 PRD 分成多个片段，每次只提取一个片段内的模块列表，最后再合并。这样单次输出的 JSON 更短，更不易出错。

### 6. 加入重试机制
- 措施：在解析失败时自动重试，最多重试 2 次。重试时可结合调整后的参数（如增大 max_tokens、降低 temperature 让输出更稳定、或使用格式约束更强的临时 prompt）。若重试仍失败，记录原始输出并中断，避免生成低质量用例。

---

## 实施建议
优先采用 **第 1 项 + 第 2 项 + 第 3 项** 的组合，通常能覆盖 90% 以上的 JSON 错误。若问题持续出现，再引入第 4、5 项优化整体鲁棒性。
